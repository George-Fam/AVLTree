Travail pratique #3 — Arbres AVL

Objectifs

  • Pratiquer l'encapsulation et concevoir des types abstraits de données.
  • Implémenter une structure de données ArbreAVL en C++.
  • Gérer la mémoire.
  • Se préoccuper de l'efficacité.

Tâches

Le TP3 consiste à :

  1. terminer l'implémentation de la classe générique ArbreAVL (fichier arbreavl.h) amorcée aux Lab6 et Lab7;
  2. préparer des tests (fichier testtp3.cpp) pour vérifier et évaluer la classe générique ArbreAVL.

Deux fichiers de départ arbreavl.h et testtp3.h sont fournis dans tp3.zip.

Interface publique

Votre implémentation finale doit offrir l'interface publique suivante. Autrement dit, tous les constructeurs, fonctions et opérateurs suivants doivent être implémentés. 

template <class T>
class ArbreAVL{
  public:
// Constructeurs
    ArbreAVL();
    ArbreAVL(const ArbreAVL&);

// Fonctions de base faites dans les labs 6-7 :
    bool vide() const;
    bool contient(const T&) const;
    void inserer(const T&);
    void vider();

// Fonctionnalités spécifiques du TP3 :
    int             taille() const; // retourne le nombre d'éléments
    
    ArbreAVL<T>&    operator = (const ArbreAVL<T>& autre);

    bool            operator == (const ArbreAVL<T>& autre) const;
    
    ArbreAVL<T>&    operator += (const ArbreAVL<T>& autre); // ajoute tous les éléments dans autre
    ArbreAVL<T>     operator +  (const ArbreAVL<T>& autre) const; // retourne l'union
    
    ArbreAVL<T>&    operator -= (const ArbreAVL<T>& autre); // retire tous les éléments dans autre
    ArbreAVL<T>     operator -  (const ArbreAVL<T>& autre) const; // retourne la différence*


private:
  ...
};

Vous pouvez ajouter d'autres fonctionnalités que celles listées ci-haut. Cependant, toutes les fonctionnalités listées ci-haut doivent être présentes, sinon les tests automatiques échoueront. À noter que plusieurs fonctionnalités des Lab6 et Lab7 ne sont pas demandées ci-haut, comme la fonction enlever et les itérateurs. Certaines d'elles pourraient être utiles pour implémenter les fonctionnalités demandées. Libre à vous de conserver ou de retirer les fonctionnalités des Lab6 et Lab7 ne sont pas demandées au TP3.

Vous pouvez modifier la représentation autant que vous le souhaitez. L'important est de respecter l'interface publique spécifiée ci-haut.

* L'opérateur - doit effectuer une différente ensembliste. En mathématique, on note la différence ensembliste avec une barre \ (backslash) et on la définie comme suit: A \ B = { x ∈ A | x ∉ B }.

Il est possible d'implémenter les opérateurs - et -= sans utiliser la fonction enlever. Indice : possible n'implique pas que ce soit une solution «parfaite»!

Tests

Contrairement au TP1, très peu de nouveaux tests sont fournis. Pour commencer, il est recommandé de réussir tous les tests pertinents des Lab6 et Lab7.

Au TP3, vous devez programmer des tests dans le programme testtp3.cpp pour vérifier les nouvelles fonctionnalités du TP3. La fonction main de testtp3.cpp doit retourner 0 si et seulement si aucune erreur n'a été détectée. Dès qu'au moins une erreur est détecté, ce programme doit retourner une valeur supérieure ou égale à 1. Il est recommandé de retourner le nombre de types d'erreurs détectées. En cas d'erreurs, votre programme doit afficher des messages permettant d'amorcer un diagnostique des erreurs détectées.

Votre programme testtp3.cpp doit aussi évaluer l'efficacité temporelle des opérations de ArbreAVL. Autrement dit, vous devez concevoir une ou plusieurs fonctions permettant de détecter des implémentations inefficaces de ArbreAVL. Le temps d'exécution sera mesuré en nombre d'opérations ou avec la commande time. Une implémentation est réputé inefficace lorsqu'un test prend significativement plus de temps par rapport à avec une implémentation efficace.

Contraintes

  1. La taille de votre programme ne doit pas dépasser 1000 lignes, 3000 mots et 20000 caractères. La taille sera mesurée avec la commande wc.
  2. Le temps d'exécution de votre programme sera limité à au moins 120 secondes (ulimit -t 120). Visez un temps d'exécution de 5-10 maximum secondes avec une implémentation efficace de ArbreAVL et sur une machine de référence (ex.: serveur java.ens.uqam.ca).
  3. La mémoire sera limitée à 2 Go (ulimit -v 2097152). Cette limite doit être largement suffisante pour tester une implémentation correcte et sans fuite de mémoire.

Perfection

Un peu comme pour le TP1, votre implémentation ArbreAVL doit viser la perfection. Lorsque vous avez différents choix possibles, au niveau de la représentation ou des algorithmes, vous devez prendre les meilleurs choix possibles. Si des compromis sont requis, vous devez prioriser, dans l'ordre, les qualités suivantes: le fonctionnement correct, l'efficacité temporelle, l'efficacité mémoire et simplicité du code.
